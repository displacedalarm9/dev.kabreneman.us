Section 1 — Purpose and Authority
M-00001 is the founding page of the MANIFEST system (class letter M). MANIFEST governs ad-hoc indexes, inventories, registries, and enumeration artifacts within UNISYS. A manifest is a structured listing of items, assets, artifacts, or states compiled for a specific operational purpose.

MANIFEST exists to formalize the common UNISYS pattern of creating inventories, checklists, indexes, and catalogs that do not belong to a specific subsystem's operational domain. While DOCSYS governs document metadata and ACCTSYS governs financial records, MANIFEST governs the indexing and enumeration layer — the "what do we have?" question across any domain.

MANIFEST is distinct from DOCSYS report indexes (RPT-), ACCTSYS ledgers (L-class), and PROJGEN's MANIFEST.json output. MANIFEST (M-class) artifacts are standalone indexes that may reference artifacts across multiple systems. PROJGEN's MANIFEST.json is a project-scoped build artifact that may be registered under M-class if it achieves operational significance beyond its originating project.

Section 2 — Manifest Identity Format
Each manifest entry receives a unique identifier following the pattern:

M-#####

Where M is the class letter and ##### is a zero-padded 5-digit sequential number. M-00001 is reserved for this founding page. Operational manifest entries begin at M-00002.

Manifest entries are DOCSYS-governed artifacts. They receive TSN assignment at creation and AUID assignment if published to the constitutional record. Manifests follow the standard publishing workflow:

TSN → AUTOENV → DOCSYS → Validate → AUID → REDOREPO → TSN.LOG

Section 3 — Versioning Status
IMPORTANT
The versioning model for MANIFEST-class artifacts is explicitly UNDETERMINED at founding. This is a deliberate architectural decision — not an oversight.
The following versioning options are under evaluation:

Option	Model	Description	Trade-offs
A	Immutable Snapshots	Each manifest is a point-in-time snapshot. Changes produce a new M-##### entry. Previous entry is obsoleted (OR-####).	Clean audit trail. High entry consumption. Simple governance.
B	Semantic Versioning	Manifests use MAJOR.MINOR.PATCH versioning per USTD-VERSION. Same M-##### ID persists across versions.	Compact identity space. Requires version history tracking. Consistent with AUTOENV/DOCSYS config.
C	Hybrid	Manifests default to immutable snapshots. The operator may designate specific manifests as "living documents" that use semantic versioning. Designation requires explicit M-class metadata flag.	Maximum flexibility. Requires per-manifest governance decision. Moderate complexity.
The operator (Kyle Breneman) retains sole authority to resolve the versioning question. Until resolved, all manifests created under M-class SHALL be treated as immutable snapshots (Option A) as the safe default. This default may be retroactively amended once the versioning decision is finalized.

Section 4 — Manifest Classification
Each manifest is classified by type. The following types are defined at founding:

Type Code	Type Name	Description	Example Use
MF-INV	Inventory Manifest	Physical or digital asset enumeration	Hardware inventory, supply lists, media library
MF-IDX	Index Manifest	Ordered reference to artifacts across systems	Master document index, template registry, crosswalk index
MF-CHK	Checklist Manifest	Ordered task or verification list with completion states	Deployment checklist, audit checklist, setup verification
MF-CAT	Catalog Manifest	Descriptive listing with classification metadata	Recipe catalog (FOODSYS), subscription catalog (ACCTSYS)
MF-REG	Registry Manifest	Authoritative enumeration of governed entities	Class letter registry, system registry, node registry
MF-PKG	Package Manifest	Contents listing for a bundled deliverable or project	PROJGEN MANIFEST.json promoted to M-class, deployment package
MF-ARC	Archive Manifest	Listing of items in cold storage or archived state	REDOREPO LASTRESORT contents, RETENSYS tier inventory
MF-ADH	Ad-Hoc Manifest	Unclassified or one-time enumeration not fitting other types	Temporary project lists, migration inventories
Section 5 — Manifest Structure Requirements
All M-class artifacts MUST contain the following structural elements:

HEADER BLOCK — Page ID, manifest type code, creation date, scope, status, author.
PURPOSE STATEMENT — One paragraph explaining why this manifest exists and what it enumerates.
ENUMERATION BODY — The listing itself. Format depends on manifest type but must use structured data (tables, numbered lists, or key-value pairs). Prose-only manifests are non-compliant.
SCOPE BOUNDARY — Explicit statement of what is included and what is excluded from the manifest.
CURRENCY STATEMENT — Date of last verification and method of update (manual, automated, or hybrid).
Optional elements:

CROSS-REFERENCES — Links to related manifests, crosswalks, or system artifacts.
COMPLETION CRITERIA — For MF-CHK type only: definition of what constitutes 100% completion.
RETENTION CLASS — REDOREPO tier assignment if the manifest has archival significance.

Section 8 — Governance Rules
Every ad-hoc index, inventory, or enumeration artifact MUST be evaluated for M-class registration. Informal lists embedded in other documents should be promoted to standalone M-class manifests when they achieve operational significance.
Manifests are DOCSYS-governed. They follow the standard artifact lifecycle: Draft → Active → Archived.
The CURRENCY STATEMENT must be updated whenever the manifest contents are verified or modified. Stale manifests (no currency update within the applicable retention period) are flagged by TRUSTSYS.
MF-REG (Registry) manifests are authoritative — they are the single source of truth for their domain. Conflicting information in other documents yields to the M-class registry.
PROJGEN MANIFEST.json files are NOT automatically M-class artifacts. Promotion requires explicit operator action (TSN assignment, DOCSYS registration, M-##### ID assignment).
Manifests may reference artifacts across any UNISYS system. Unlike crosswalks (which map relationships), manifests enumerate items.
The distinction between MANIFEST and CROSSWALK is: CROSSWALK (X) answers "how do systems connect?" while MANIFEST (M) answers "what exists?"
Until the versioning question (Section 3) is resolved, all M-class artifacts default to immutable snapshot behavior.
